1. Continue-batching：持续的向GPU中发送请求数据，而不是离散的进行数据推理，一个请求结束立刻下发新的请求，保证GPU利用率。它不再以“整个请求”为单位，而是以“每一次生成Token的迭代”为单位进行调度，不再强制将所有序列补齐成整齐的矩形，而是将所有请求的Token在内存中直接拼接成一维数组

   - Static-batching 缺陷：
     - 将多个请求打包为一个batch处理，由于每个请求生成的文本长度不一致，短的会很快生成完毕，导致“木桶效应”；导致GPU在处理完短请求后，会有一段空闲等待的时间，造成资源浪费
     - 由于一个batch内味了让长短不一致的请求并行处理，使用padding技术，导致GPU花费时间计算无意义的padding内容，造成算力浪费

2. Paged Attention：

   - 将 KV Cache 切分为固定大小的“页”（例如每页存 16 或 32 个 token 的数据）。这些页不需要在显存中连续存放，它们可以散落在显存的任何角落
   - 系统维护一个块表，用来记录每个请求的逻辑页在物理显存中的位置
   - 当请求刚开始时，系统只分配少量的页。随着生成的文本变长，系统再动态地申请新的空闲页，并将其链接到块表中
   - 写时复制机制

   - 解决连续内存分配的问题：
     - 过度预留导致的内存浪费，为防止显存溢出，系统会按最大可能长度给每个请求预留显存
     - 内存碎片化

3. Copy-on-write：对于共享同一个 Prompt 的多个请求，可以让它们的块表指向同一块物理显存。只有当某个请求需要修改数据时，系统才会复制一份新的页给它

4. Radix Attetnion：

   - 解决的问题：传统方式通常只能识别同一个请求内部的连续性。一旦请求结束，或者面对新的请求，哪怕新请求的前缀和刚才的一模一样，系统也会“失忆”，重新把那些 Token 算一遍
   - 全局共享的 KV 缓存树：树的每一条路径代表一段 Token 序列，每个节点存储对应的 KV Cache 数据
   - 当一个新的请求到来时，系统不会急着计算，而是先去这棵树上“查找”，一旦发现路径上的节点已经存在（比如之前的对话历史或相同的 System Prompt），系统就直接复用这些节点上已经计算好的 KV Cache，只有在树上找不到路径的“分叉点”之后，系统才开始进行新的计算，并将新计算的节点“嫁接”到树上
   - 使用最近最少使用（LRU）等策略，递归地修剪树的叶子节点

5. PD分离部署：模型计算过程拆成两步:P（prefill，算力密集）、D（decode，存储密集），将P和D拆分部署提升GPU利用率

6. Attention与FFN分离：将Attention与FFN子模块分别部署到不同设备上，实现较高推理性能

7. Auto Prefix Cache 自动前缀缓存：复用历史的KV cache信息，降低prefill阶段的计算量从而提升首token性能

   - 解决的问题：传统推理框架通常把每个请求都当作全新的任务来处理。即使新请求的开头和之前的请求一模一样，模型也会“失忆”，把那些重复的 Token 再重新计算一遍 KV Cache
   - 监控所有请求，一旦发现某个文本前缀已经计算过，就会把它对应的 KV Cache 结果缓存起来。当下一个请求到来时，它会先检查这个请求的开头是否命中了缓存
     - 当一个请求进来时，系统会计算其前缀的哈希值
     - 构建在 PagedAttention 的内存管理机制之上
     - 使用最近最少使用（LRU）等策略
